## 1. 렌더링(Rendering)이란?

### 정의

> **렌더링 = "그리는 행위" 그 자체**

웹에서 "렌더링"이라는 단어가 보이면 무조건 **"그린다"**고 이해하면 됩니다.

- 3D 게임에서도 렌더링 = 3D 그래픽을 화면에 그리는 행위
- 웹에서도 렌더링 = HTML/CSS를 화면에 그리는 행위
- JS가 실행되면서 HTML을 브라우저에 그리는 것도 렌더링

### 정리

```
렌더링 = 그리는 행위
HTML   = 그 결과물 (그림)
JS     = 그리는 화가
브라우저 = 그림이 걸리는 전시관
```

---

## 2. 정적 렌더링 vs 동적 렌더링

### 정적 렌더링 (Static Rendering)

> 서버가 **미리 완성된 HTML 파일**을 만들어서 그대로 브라우저에 보내는 방식

- 이미 다 그려진 그림을 배달해주는 것
- 누가 접속해도 똑같은 HTML을 받음
- 변하지 않는 콘텐츠에 적합 (회사 소개 페이지 등)

### 동적 렌더링 (Dynamic Rendering)

> 브라우저가 JS를 실행하면서 **접속한 사용자에 맞게 실시간으로 화면을 구성**하는 방식

- 그림 그리는 방법(JS)을 보내고, 브라우저가 직접 그림
- 사용자마다 다른 화면을 보여줄 수 있음
- 우리 프로젝트가 사용하는 방식

### 한눈에 비교

| 구분 | 누가 그리나? | 언제 그리나? | 특징 |
|------|-------------|-------------|------|
| 정적 렌더링 | 서버 | 배달 전에 미리 | SSR에서 사용 |
| 동적 렌더링 | 브라우저(클라이언트) | 접속 후 실시간 | CSR에서 사용 |

## 3. CSR vs SSR — 렌더링 방식의 차이

### SSR (Server-Side Rendering) — 옛날 방식

```
서버가 그림을 다 그려서 → 브라우저에게 완성된 HTML 전달
```

- 서버가 화가 역할
- 처음부터 완성된 화면이 보임
- 서버 부하가 높음

### CSR (Client-Side Rendering) — 현재 우리 방식

```
서버는 도화지(빈 HTML) + 붓과 그리는 방법(JS)만 전달
→ 브라우저(클라이언트)가 직접 화면을 그림
```

- 브라우저가 화가 역할
- 처음에는 빈 화면이었다가 JS가 실행되면서 화면 완성
- 서버 부하 낮음, 이후 인터랙션이 빠름

### 우리 프로젝트가 CSR인 이유

- 서버 리소스 절약
- 빠른 인터랙션 (데이터 변경 시 전체 페이지 재로딩 없음)
- 현대 웹 개발의 표준

### CSR 실행 흐름 전체

```
1. 브라우저가 빈 HTML + JS 보따리 수신 (서버에서 배달)
        ↓
2. 브라우저가 JS 실행 시작
        ↓
3. 스켈레톤 UI (회색 뼈대) 즉시 표시
        ↓
4. 동시에 API 요청 → 실제 데이터 요청 시작
        ↓
5. 데이터 도착 → 회색 뼈대에 실제 데이터 채워 넣음
        ↓
6. 화면 완성!
```

## 4. 스켈레톤 UI (Skeleton UI)

### 왜 필요한가?

CSR 방식에서는 처음에 빈 화면이 보입니다.

```
브라우저가 JS 받음 → JS 실행 → API 요청 → 데이터 도착 → 화면 완성
                               ↑
                          이 시간 동안 화면이 비어 있음!
```

사용자 입장에서 빈 화면을 보면 → "사이트 고장났나?" → 이탈

### 스켈레톤 UI란?

> 데이터가 오기 전까지 **콘텐츠의 회색 뼈대(윤곽)**를 먼저 보여주는 기법

실제로 본 적 있는 예시:
- 인스타그램 피드/스토리 갤러리 로딩 시 → 사진 대신 회색 네모가 먼저 뜸
- 페이스북 피드 로딩 시 → 글자 대신 회색 줄이 먼저 뜸

### 동작 순서

```
1. 접속 → 빈 HTML + JS 보따리 도착
2. 브라우저가 JS 실행 시작
3. 스켈레톤 UI (회색 뼈대) 즉시 표시
4. 동시에 API 요청으로 실제 데이터 받아오기 시작
5. 데이터 도착 → 회색 뼈대에 실제 데이터 채워 넣음
6. 화면 완성!
```

### 코드 작성 시 반드시 함께 작성해야 하는 것

스켈레톤 UI를 만들 때는 **두 가지를 세트로** 작성해야 합니다.

| 코드 | 역할 |
|------|------|
| **스켈레톤 UI 코드** | 회색 뼈대를 그리는 부분 |
| **API 통신 코드** | 데이터를 가져오는 부분 (try-catch + API 주소 + 함수) |

> 핵심 로직은 `try-catch`를 이용한 API 요청 코드 5~10줄이고,
> 나머지는 보안/에러 처리 코드입니다.
> 처음에는 핵심 로직만 작성해도 동작하며, 이후 예외 처리를 추가해 나갑니다.

### 페이지 전환도 가능

데이터를 스켈레톤에 채우는 것 외에도, 아예 **다른 페이지로 이동**할 수도 있습니다.

```
방법 1: 스켈레톤 → 데이터 채움 → 같은 페이지 완성
방법 2: 로그인 완료 → 메인 피드 페이지로 이동 (라우터 사용)
```

```javascript
// 예시: 로그인 성공 후 메인 페이지로 이동
router.replace('/main')
```

## 5. State(상태) 관리 기초

### State(상태)란?

> 웹 앱이 실행되는 동안 브라우저가 **기억하고 유지해야 하는 데이터**

### 예시

- 현재 로그인한 유저 정보
- 지금 선택된 탭이 무엇인지
- 입력창에 작성 중인 텍스트

### 저장 위치에 따른 구분

| 위치 | 특징 | 사용 예 |
|------|------|---------|
| **메모리 (변수)** | 새로고침하면 사라짐, 속도 빠름 | 현재 사용자 정보 |
| **LocalStorage** | 브라우저를 꺼도 남아 있음 | 사용자 설정값 |
| **SessionStorage** | 탭을 닫으면 사라짐 | 임시 데이터 |

### React에서 State를 관리하는 도구들

| 도구 | 용도 |
|------|------|
| `useState` | React 기본 제공 (컴포넌트 내부 상태) |
| `Recoil`, `Zustand` | 전체 앱에서 상태를 공유할 때 사용 |

## 6. Toast 알림 라이브러리

### Toast란?

화면 하단에 잠시 떴다가 사라지는 **가벼운 알림 메시지**입니다.

예시:
- "로그인 되었습니다 ✅"
- "서버에 연결할 수 없습니다 ❌"
- "이미지가 업로드되었습니다 ✅"

### AI에게 코드 요청 시

```
"우리는 Toast 라이브러리를 쓰고 있어"
→ AI가 자동으로 Toast 알림을 적용해서 코드를 짜줌
```

## 7. 타임아웃 (Timeout)

### 왜 필요한가?

API 요청을 보냈는데 서버가 먹통이면 → 사용자가 무한정 기다려야 합니다.

### 타임아웃 설정

- **기본값**: 브라우저 기본 타임아웃 = 약 60초
- **권장**: 상황에 맞게 직접 설정 (예: 10초)

```
10초 동안 응답이 없으면 → "서버 응답이 없습니다" 에러 처리
```

타임아웃도 인터셉터에서 자동 처리할 수 있습니다.

## 8. 예외 처리의 중요성

### 수업에서 강조한 핵심 원칙

> **"완성도는 오류와 예외 처리에서 나온다"** — 노션 메인 페이지에도 적혀 있는 핵심 원칙

### 핵심 로직 vs 예외 처리 코드 비율

```
실제 통신 핵심 로직:  try { ... } 약 5~10줄
에러/보안 처리 코드: 나머지 80~90%
```

핵심 로직보다 **예외 처리 코드가 훨씬 많습니다.** 이게 정상입니다.

### 예외 처리 예시

| 상황 | 예외 처리 |
|------|-----------|
| 서버 응답 없음 | 타임아웃 후 "서버 오류" Toast 표시 |
| 토큰 만료 (401) | 자동으로 Silent Refresh 수행 |
| 잘못된 파일 업로드 | "PNG, JPG만 가능합니다" 즉시 안내 |
| 네트워크 끊김 | "네트워크를 확인해주세요" 안내 |
| 권한 없음 (403) | "접근 권한이 없습니다" Toast 표시 |

### 개발 순서 권장

```
1단계: 핵심 로직만 작성 (try-catch로 기본 통신만)
2단계: 동작 확인
3단계: 예외 처리 하나씩 추가 (에러 케이스를 발견할 때마다)
```

## 9. 최종 요약 — 프론트엔드 전체 흐름

### 우리 프로젝트의 프론트엔드가 하는 일

```
1. JWT 신분증 관리
   - Access Token + Refresh Token → 둘 다 HttpOnly 쿠키에 저장
   - 인터셉터로 모든 API 요청에 자동으로 신분증 첨부

2. 유효성 검증
   - 사용자 입력값을 서버 보내기 전에 미리 검증

3. CSR 기반 화면 렌더링
   - JSX/TSX로 코드 작성 → NPM/Vite가 JS로 번역
   - 처음 접속 시 스켈레톤 UI 표시
   - API로 데이터 받아와서 실제 화면 완성

4. Silent Refresh 처리
   - 401 에러 감지 → Refresh Token으로 자동 재발급 → 재시도
   - 사용자는 전혀 모르게 처리됨
```

### 코드 작성 시 체크리스트

- [ ] 스켈레톤 UI 컴포넌트 작성
- [ ] API 통신 함수 작성 (try-catch 포함)
- [ ] 에러 처리 (Toast 알림)
- [ ] 타임아웃 설정
- [ ] 인터셉터 활용 확인
- [ ] 유효성 검증 추가

### AI에게 코드 요청 시 꼭 언급할 것

```
- "우리는 Axios 인터셉터 사용해"
- "우리는 Toast 라이브러리 써"
- "우리는 CSR 방식이야"
- "JWT 토큰은 Access/Refresh 둘 다 HttpOnly 쿠키에 저장해"
- "401이면 Silent Refresh 처리해줘"
```

### Q3. CSR과 SSR의 차이를 설명해보세요.

<details>
<summary>▶ 정답 보기</summary>

- **SSR** — 서버가 그림을 다 그려서 완성된 HTML을 브라우저에게 전달
- **CSR** — 서버는 빈 도화지(HTML)와 그리는 방법(JS)만 전달하고, **브라우저**가 직접 그림

우리 프로젝트는 **CSR** 방식을 사용합니다.
이유: 서버 리소스 절약, 빠른 인터랙션, 현대 웹 개발의 표준

</details>

### Q4. 스켈레톤 UI가 왜 필요하고, 동작 순서를 설명해보세요.

<details>
<summary>▶ 정답 보기</summary>

**왜 필요한가?**
CSR 방식은 브라우저가 직접 화면을 그리기 때문에,
서버에서 데이터를 받아오는 동안 화면이 비어 있습니다.
빈 화면을 보면 사용자가 "고장났나?" 하고 이탈할 수 있기 때문입니다.

**동작 순서:**
1. JS 보따리 도착 및 실행 시작
2. 스켈레톤 UI (회색 뼈대) 즉시 표시
3. 동시에 API로 실제 데이터 요청
4. 데이터 도착 → 회색 뼈대 자리에 실제 데이터 채워 넣음
5. 화면 완성

**실생활 예시:** 인스타그램 피드 로딩 시 보이는 회색 네모/줄 영역

</details>

### Q7. "완성도는 오류와 예외 처리에서 나온다"는 말의 의미를 코드 비율로 설명해보세요.

<details>
<summary>▶ 정답 보기</summary>

실제 API 통신 코드의 구성:

```
핵심 통신 로직:  try { ... } → 전체의 10~20%
에러/보안 처리:  나머지 → 전체의 80~90%
```

핵심 로직보다 **예외 처리 코드가 훨씬 많은 게 정상**입니다.
처음에는 핵심 로직만 짜도 동작하지만, 완성도(서비스 품질)는 예외 처리가 결정합니다.

</details>

### Q8. (종합) 사용자가 웹사이트에 처음 접속해서 프로필 화면을 보는 전체 흐름을 처음부터 끝까지 설명해보세요.

<details>
<summary>▶ 정답 보기</summary>

```
1. 사용자가 브라우저 주소창에 URL 입력

2. 서버가 빈 HTML + JS 보따리 전송 (CSR 방식)

3. 브라우저가 JS 실행 시작
   → 스켈레톤 UI (회색 뼈대) 즉시 표시

4. 인터셉터가 자동으로 Access Token을 헤더에 붙여서
   프로필 API 요청 전송

5-A. Access Token이 유효한 경우:
   → 서버가 프로필 데이터 응답 (200)
   → 스켈레톤 자리에 실제 데이터 채워짐
   → 화면 완성!

5-B. Access Token이 만료된 경우 (401):
   → 인터셉터가 401 감지
   → Refresh Token으로 자동 재발급 요청 (Silent Refresh)
   → 새 Access Token 받음
   → 원래 프로필 API 요청 재시도
   → 사용자는 이 과정을 전혀 모름
   → 화면 완성!
```

</details>
