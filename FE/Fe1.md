## 1. JWT — 디지털 신분증

### JWT가 뭔가요?

**JWT (JSON Web Token)** 는 서버가 발급해주는 **디지털 신분증**입니다.

### 동작 흐름

```
사용자가 카카오 로그인
        ↓
카카오가 인증 완료 → 우리 DB에 사용자 저장
        ↓
우리 서버(백엔드)가 JWT 신분증을 발급해서 프론트에 전달
        ↓
프론트는 이 신분증을 안전하게 저장해두고
모든 API 요청 시마다 이 신분증을 헤더에 실어서 보냄
```

### JWT의 형태

- JSON 형태로 만들어진 웹 토큰
- 사람으로 치면 주민등록증처럼 "이 사람이 누구인지"를 증명하는 역할

## 2. Access Token vs Refresh Token

JWT 신분증은 두 가지 종류가 있습니다.

| 구분 | Access Token | Refresh Token |
|------|-------------|---------------|
| **비유** | 놀이공원 자유이용권 팔찌 | 은행 인감도장 |
| **유효기간** | 짧음 (60분, 실제 서비스는 15~30분) | 길음 (14일) |
| **저장 위치** | HttpOnly 쿠키 (우리 프로젝트 기준) | HttpOnly + Secure + SameSite 쿠키 |
| **역할** | 실제 서비스 이용 시 제시하는 신분증 | Access Token이 만료됐을 때 재발급 받기 위한 열쇠 |

> ⚠️ **수업 중 정정된 내용**
> 처음에는 "Access Token은 메모리, Refresh Token은 쿠키"라고 설명했지만,
> 우리 프로젝트는 **인터셉터가 쿠키에서 자동으로 꺼내 보내주기 때문에**
> **둘 다 HttpOnly 쿠키에 저장**하는 방식으로 구현되어 있습니다.

### 왜 두 가지로 나누나요?

- Access Token만 있으면: 만료될 때마다 사용자가 매번 다시 로그인해야 함 → 불편
- Refresh Token이 있으면: 사용자 모르게 조용히 Access Token을 갱신 → 편리

### 삼중 보안 (쿠키 보안 옵션 3가지)

| 옵션 | 역할 | 방어 대상 |
|------|------|-----------|
| **HttpOnly** | JS가 쿠키를 읽지 못하게 막음 | XSS 공격 |
| **SameSite** | 우리 서비스 도메인에서만 쿠키 전송 | CSRF 공격 |
| **Secure** | HTTPS에서만 쿠키 전송 | 네트워크 도청 |

> ⚠️ `Secure` 옵션은 HTTPS 환경에서만 작동합니다. 배포 시 HTTPS 설정 필요.

## 3. 토큰 재발급 흐름 (Silent Refresh)

사용자는 전혀 모르게, 뒤에서 자동으로 신분증을 갱신하는 과정입니다.

### 단계별 흐름

```
1단계: [FE] Access Token을 헤더에 실어 API 요청 전송
           ↓
2단계: [BE] "이 신분증 만료됐는데요" → 401 Unauthorized 응답
           ↓
3단계: [FE] "아, 만료됐구나" → 재발급 API 호출
       (이때 쿠키에 저장된 Refresh Token을 브라우저가 자동으로 서버에 전송)
           ↓
4단계: [BE] Refresh Token 확인 후 → 새 Access Token 발급해서 응답
           ↓
5단계: [FE] 새 Access Token을 쿠키에 갱신 후 → 원래 요청 재시도
           ↓
       사용자는 결과만 받음 (중간 과정을 전혀 모름)
```

### 왜 "Silent(조용한)" Refresh인가?

- 사용자는 로그인이 갱신되는 걸 전혀 모릅니다
- 뒤에서 인터셉터가 만료된 팔찌를 몰래 새것으로 바꿔주기 때문
- "조용한(Silent) 갱신(Refresh)" → **Silent Refresh**

### 중요 포인트

> 이 재발급 흐름은 사용자에게 보여주면 안 됩니다.
> 프론트엔드에서 자동으로 처리해야 하며, 사용자는 오직 "요청 → 결과"만 알면 됩니다.

## 4. 유효성 검증 (Validation)

### 왜 프론트에서도 검증하나요?

백엔드에서도 검증하지만, 프론트에서 **미리** 걸러주는 이유가 있습니다.

| 이유 | 설명 |
|------|------|
| **서버 부하 감소** | 잘못된 데이터가 서버까지 가서 처리되는 리소스 낭비 방지 |
| **통신 오버헤드 제거** | HTTP 요청 자체가 헤더 + 통신 + 처리 + 응답 비용이 있음 |
| **빠른 피드백** | 사용자에게 즉시 오류를 알려줄 수 있음 |

### 어떤 걸 검증하나요?

- **타입 검증**: 숫자를 넣어야 하는 곳에 문자열을 넣는 경우 차단
- **파일 형식 검증**: 이미지 업로드 시 `.png`, `.jpg`가 아닌 `.txt`, `.md` 파일 차단
- **필수값 검증**: 비어 있으면 안 되는 필드가 비어 있는 경우 차단

### 흐름

```
사용자 입력
    ↓
[FE] 1차 유효성 검증 → 잘못됐으면 즉시 오류 표시
    ↓ (통과)
[BE] 2차 유효성 검증 → 교차 검증
    ↓ (통과)
DB 저장 및 처리
```

## 5. API란 무엇인가

### 핵심 정의

> **API = 주소 (URL)**
> 백엔드에 데이터를 요청하거나 기능을 사용하기 위한 특정 주소

### 식당 비유로 이해하기

| 역할 | 비유 |
|------|------|
| 사용자 | 손님 |
| 프론트엔드 | 웨이터 |
| API | 주문 (어디로 보낼지 = 주소) |
| 헤더 | 신분증 |
| 바디(Body) | 주문 내용물 |
| 백엔드 서버 | 주방 주인 |
| DB | 재료 창고 |

### 예시 흐름

```
손님: "술 주세요"
웨이터(FE): 주방(BE)에 술 달라고 요청
    + 신분증(JWT)을 헤더에 같이 제출
주방(BE): 신분증 확인 → DB에서 술 꺼내옴 → 웨이터에게 전달
웨이터(FE): 손님에게 술 전달
```

## 6. HTTP 요청의 3대 요소

API 요청은 단순히 주소를 부르는 것이 아니라 **세 가지 요소로 구성된 우편물**을 보내는 것입니다.

### 1) Method (우편물 종류)

이 요청이 **무엇을 하려는지** 결정합니다.

| Method | 의미 | 예시 |
|--------|------|------|
| `GET` | 데이터 조회 | 프로필 정보 가져오기 |
| `POST` | 데이터 생성/전송 | 회원가입, 로그인 |
| `PUT` / `PATCH` | 데이터 수정 | 프로필 수정 |
| `DELETE` | 데이터 삭제 | 계정 삭제 |

### 2) Headers (봉투 겉면)

요청에 대한 **부가 정보(메타데이터)**를 담습니다.

```json
{
  "Authorization": "Bearer eyJhbGciOiJI...",
  "Content-Type": "application/json"
}
```

- JWT 신분증이 여기 `Authorization` 헤더에 들어갑니다
- 프론트엔드의 핵심 영역

### 3) Body (편지 내용물)

백엔드에 **실제로 전달할 데이터**입니다.

```json
{
  "username": "주원",
  "email": "juwon@example.com"
}
```

- GET 요청은 보통 Body가 없음
- POST, PUT 등에서 실제 데이터를 담아 전송

## 7. 신분증 보관과 제시 방법

### 보관 — 어디에 저장하나?

| 저장소 | 특징 | 보안 수준 |
|--------|------|-----------|
| LocalStorage | 누구나 JS로 접근 가능 | ⚠️ 낮음 |
| SessionStorage | 탭 닫으면 사라짐 | ⚠️ 보통 |
| Secure Storage | 암호화 저장, 쉽게 열람 불가 | ✅ 높음 |
| HttpOnly Cookie | JS로 읽기 불가능 | ✅✅ 매우 높음 |

> ✅ **우리 프로젝트: Access Token + Refresh Token 둘 다 HttpOnly Cookie에 저장**
> 인터셉터가 쿠키에서 자동으로 토큰을 꺼내서 헤더에 붙여주기 때문에 둘 다 쿠키 사용

### 제시 — 어떻게 서버에 보내나?

모든 API 요청 시 HTTP 헤더의 `Authorization` 필드에 토큰을 실어서 전송합니다.

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

## 8. 인터셉터 (Interceptor)

### 인터셉터란?

> 모든 API 요청이 나가기 **직전**, 자동으로 쿠키에서 신분증을 꺼내 헤더에 붙여주는 **자동 장치**

### 왜 필요한가?

매번 API 요청 코드마다 직접 토큰을 꺼내 헤더에 붙이면:
- 코드가 반복되고 지저분해짐
- 실수로 빼먹을 수 있음
- 토큰 만료 처리를 매번 따로 해야 함

### 인터셉터가 하는 일

1. 요청 보내기 전 → 자동으로 Access Token을 헤더에 추가
2. 401 에러 받으면 → 자동으로 토큰 재발급 요청 (Silent Refresh)
3. 재발급 후 → 자동으로 원래 요청 재시도

### 사용 라이브러리

우리 프로젝트는 **Axios** 라이브러리의 인터셉터 기능을 사용합니다.

```
AI에게 코드 요청 시: "우리는 Axios 인터셉터를 사용하고 있어"
→ AI가 자동으로 인터셉터 적용해서 코드를 짜줌
```

## 9. 브라우저 vs 프론트엔드 코드

많은 사람이 헷갈리는 개념입니다. 둘은 다릅니다.

### 브라우저 (실행 환경)

- Chrome, Safari, Firefox 등
- 코드가 **실행되는 공간** = 무대, 엔진
- HTML/CSS를 그리는 **렌더링 엔진** 보유
- JavaScript를 실행하는 **JS 엔진** 보유
- 저장소(LocalStorage), 위치정보 등 **Web API** 제공

### 프론트엔드 코드 (프로그램)

- 우리가 직접 작성하는 `.tsx`, `.jsx`, `.js` 파일들
- 브라우저 위에서 실행되는 **실제 공연(프로그램)**
- 사용자의 클릭, 입력, 이미지 업로드 등 **이벤트를 처리**
- 데이터를 받아서 **예쁜 화면으로 변환**하는 역할

### 비유

```
브라우저 = 극장 (공연이 펼쳐지는 공간)
프론트엔드 코드 = 극본 + 배우 (실제 공연 내용)
```

## 10. JSX / TSX와 빌드 도구(NPM)

### JS / HTML / CSS의 관계

```
HTML = 이미 그려진 그림 (도화지)
CSS  = 그림의 색깔과 스타일
JS   = 화가 (HTML을 실시간으로 그려주는 존재)
```

> JS가 실행되면서 브라우저에 HTML을 직접 그려냅니다.

### JSX / TSX가 뭔가요?

- **JSX** = JavaScript + HTML을 합쳐놓은 파일 형식 (`.jsx`)
- **TSX** = TypeScript + HTML을 합쳐놓은 파일 형식 (`.tsx`)
- 개발자가 **편하게 작성**하기 위해 만들어진 형식

### 문제: 브라우저는 JSX/TSX를 모른다

```
브라우저가 이해하는 언어: HTML, CSS, JS (순수)
브라우저가 이해 못하는 언어: JSX, TSX → 외계어!
```

### 해결: 빌드 도구가 번역해준다

```
우리가 작성: .tsx / .jsx (편한 형태)
        ↓
NPM + 빌드 도구(Vite 등)가 자동 번역
        ↓
브라우저가 읽는: .js + .html (순수 형태)
```

### NPM 명령어 정리

| 명령어 | 역할 |
|--------|------|
| `npm install` | 프로젝트에 필요한 라이브러리(빌드 도구 포함) 설치 |
| `npm run dev` | 개발 서버 실행 (코드 번역 + 로컬에서 화면 띄움) |

### 참고 — 백엔드 개발 서버 명령어 비교

| 명령어 | 역할 |
|--------|------|
| `npm run dev` | 프론트엔드 개발 서버 실행 |
| `fastapi dev` | 백엔드 개발 서버 실행 (`uvicorn run app.main:app`과 동일) |

`fastapi dev`는 프로젝트 내 `app.main`의 `app` 객체를 찾아서 실행합니다.
`uvicorn`을 직접 쓰는 것과 결과는 같고, 명령어만 더 짧고 간단합니다.

## 11. 보안의 트레이드오프

### 트레이드오프(Trade-off)란?

> 하나를 높이면 하나가 낮아지는 반비례 관계

| 높이면 | 낮아지는 것 |
|--------|------------|
| 보안 강도 | 속도, 리소스 효율 |
| 편의성 | 보안 강도 |
| 코드 복잡도 | 유지보수 용이성 |

### JWT 사용에서의 트레이드오프

| 방식 | 장점 | 단점 |
|------|------|------|
| JWT (현재 방식) | 빠름, 서버 리소스 절약 | 토큰 탈취 위험 존재 |
| DB 토큰 저장 | 완벽한 보안 (프론트에서 토큰 볼 수 없음) | 서버 리소스 많이 소모, 느림 |

### 보안에 대한 올바른 관점

> 완벽한 보안은 없습니다.
> 보안의 핵심은 **"침입 방지"**가 아니라 **"빠른 탐지와 빠른 대응"**입니다.

- 뚫린다는 걸 항상 가정하고 설계
- 뚫렸을 때 **어디서 뚫렸는지 탐지**하는 시스템 구축
- 탐지되면 **빠르게 대응**하는 프로세스 준비

> 보안이 복잡해질수록 빈틈도 많아집니다. 적절한 수준의 균형이 중요합니다.

## 12. ✅ 이해도 확인 Q&A (1부)

### Q1. JWT가 무엇인지 한 문장으로 설명해보세요.

<details>
<summary>▶ 정답 보기</summary>

**JWT(JSON Web Token)** 는 로그인 성공 후 백엔드가 발급해주는 **디지털 신분증**으로,
이후 모든 API 요청 시 헤더에 실어 보내서 "내가 누구인지"를 증명하는 데 사용합니다.

</details>

---

### Q2. Access Token과 Refresh Token의 차이점을 설명해보세요. (유효기간, 저장 위치, 역할)

<details>
<summary>▶ 정답 보기</summary>

| | Access Token | Refresh Token |
|--|--|--|
| 유효기간 | 짧음 (60분, 실제 서비스 15~30분) | 길음 (14일) |
| 저장 위치 | HttpOnly 쿠키 (우리 프로젝트 기준) | HttpOnly 쿠키 |
| 역할 | 실제 서비스 이용 시 제시하는 신분증 | Access Token 만료 시 재발급받기 위한 열쇠 |

> ⚠️ 일반론은 "Access Token = 메모리"이지만, 우리 프로젝트는 **둘 다 HttpOnly 쿠키**입니다.

**비유:** Access Token = 놀이공원 자유이용권 팔찌 / Refresh Token = 인감도장

</details>

---

### Q3. Silent Refresh 흐름을 순서대로 설명해보세요.

<details>
<summary>▶ 정답 보기</summary>

1. FE가 Access Token을 헤더에 실어 API 요청
2. BE가 "신분증 만료됨" → **401 에러** 응답
3. FE가 재발급 API 호출 (쿠키의 Refresh Token이 브라우저에 의해 자동 전송)
4. BE가 Refresh Token 확인 후 → 새 Access Token 발급
5. FE가 새 토큰을 **쿠키에 갱신** 후 → 원래 요청 **재시도**
6. 사용자는 이 모든 과정을 전혀 모름 → "조용한(Silent) 갱신"

</details>

---

### Q4. 프론트에서도 유효성 검증을 하는 이유 3가지를 말해보세요.

<details>
<summary>▶ 정답 보기</summary>

1. **서버 부하 감소** — 잘못된 데이터가 서버까지 가서 처리되는 리소스 낭비 방지
2. **통신 오버헤드 제거** — HTTP 요청 자체가 비용(헤더, 통신, 처리, 응답)이 들기 때문
3. **빠른 피드백** — 사용자에게 즉시 오류를 알려줄 수 있음

백엔드도 검증하므로 **교차 검증** 구조가 됩니다.

</details>

### Q5. HTTP 요청의 3대 요소를 설명하고, JWT는 어디에 들어가나요?

<details>
<summary>▶ 정답 보기</summary>

1. **Method** — 이 요청이 무엇을 할지 (GET/POST/PUT/DELETE)
2. **Headers** — 요청의 부가 정보 — **JWT 신분증이 `Authorization` 필드에 들어갑니다**
3. **Body** — 실제 전달할 데이터 내용물

```
Authorization: Bearer eyJhbGciOiJI...
```

</details>

### Q6. 인터셉터(Interceptor)가 하는 일 3가지를 말해보세요.

<details>
<summary>▶ 정답 보기</summary>

1. **요청 전** — 자동으로 Access Token을 헤더에 추가
2. **401 에러 감지 시** — 자동으로 Refresh Token으로 재발급 요청 (Silent Refresh)
3. **재발급 후** — 실패했던 원래 요청을 자동으로 재시도

우리 프로젝트는 **Axios** 라이브러리의 인터셉터 기능을 사용합니다.

</details>

### Q7. JSX/TSX 파일을 브라우저가 바로 실행하지 못하는 이유와 해결 방법은?

<details>
<summary>▶ 정답 보기</summary>

**이유:** 브라우저는 순수한 HTML, CSS, JS만 이해합니다. JSX/TSX는 브라우저 입장에서 외계어입니다.

**해결:** `npm install`로 빌드 도구(Vite 등)를 설치하고, `npm run dev`를 실행하면
빌드 도구가 JSX/TSX를 브라우저가 읽을 수 있는 JS + HTML로 자동 번역해줍니다.

```
.tsx / .jsx → (NPM + Vite 번역) → .js + .html
```

</details>

### Q8. Refresh Token 쿠키의 삼중 보안 3가지와 각각의 역할을 설명해보세요.

<details>
<summary>▶ 정답 보기</summary>

1. **HttpOnly** — JavaScript가 쿠키를 읽지 못하게 막음 → XSS 공격 방어
2. **SameSite** — 우리 서비스 도메인에서만 쿠키 전송 → CSRF 공격 방어
3. **Secure** — HTTPS에서만 쿠키 전송 → 네트워크 도청 방어

⚠️ `Secure`는 HTTPS 환경에서만 작동합니다.

</details>

### Q9. 보안의 트레이드오프란 무엇이고, 우리가 JWT를 쓰는 이유는?

<details>
<summary>▶ 정답 보기</summary>

**트레이드오프** = 하나를 높이면 하나가 낮아지는 반비례 관계

- 보안을 높이면 → 속도/효율이 낮아짐 (예: 은행 앱이 느린 이유)
- 편의성을 높이면 → 보안이 약해짐

**JWT를 쓰는 이유:**
- DB 토큰 저장이 더 안전하지만 서버 리소스 소모가 크고 느림
- JWT는 빠르고 서버 리소스를 절약하면서 삼중 보안으로 적절한 균형 유지

> 핵심: **완벽한 보안은 없습니다.** 보안의 목표는 "침입 방지"가 아니라 **"빠른 탐지와 빠른 대응"**입니다.

</details>

### Q10. (종합) Access Token이 만료됐을 때 처음부터 끝까지 어떤 일이 일어나는지 설명해보세요.

<details>
<summary>▶ 정답 보기</summary>

```
1. FE가 API 요청 (인터셉터가 자동으로 Access Token을 헤더에 첨부)
2. BE → 401 응답 (Access Token 만료)
3. 인터셉터가 401 감지
4. 재발급 API 호출 → 쿠키의 Refresh Token이 자동으로 BE에 전송
5. BE가 Refresh Token 유효성 확인 → 새 Access Token 발급
6. FE가 새 Access Token을 쿠키에 갱신
7. 원래 요청 재시도 → 성공
8. 사용자는 이 과정을 전혀 모름
```

만약 Refresh Token도 만료됐다면 → 로그인 페이지로 이동

</details>
